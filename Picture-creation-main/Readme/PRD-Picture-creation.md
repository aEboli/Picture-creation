# PRD-Picture-creation

## 1. 一句话概述

`Picture-creation` 是一个面向电商团队的本地化 AI 生图工作台，用于把“上传原图 / 组织参考图 / 批量生成 / 审核导出 / Windows 交付”收口到同一个桌面化工作流里。

## 2. 背景

电商团队在主图、详情页、活动图、A+ 图和多市场适配中，常见痛点是：

- Prompt、模板、品牌规则分散，难以复用。
- 同一套参数既需要“多图联合创作”，也需要“批量套同模板”。
- 成果审核、历史回看、导出与协作同步缺乏闭环。
- 交付仍依赖开发环境，安装、打包和迁移成本高。

`v0.7.0` 的核心目标，是把产品从“可交付”推进到“可命名、可打包、可仓库化交接”的稳定版本。

## 3. 目标用户

| 用户 | 诉求 |
| --- | --- |
| 电商运营 | 快速批量生成可上架、可投放的商品素材 |
| 设计支持 | 复用模板与品牌规则，减少重复劳动 |
| 商品团队 | 按品类、平台、市场稳定产出成套图片 |
| 维护者 | 能打包、能装机、能迁移、能定位问题 |

## 4. 产品目标

### 4.1 交付目标

- 统一产品命名为 `Picture-creation`。
- 输出标准化 README、使用说明、PRD、版本说明。
- 新建独立 GitHub 仓库承载该项目快照与后续迭代。

### 4.2 功能目标

- 保留五种创建模式。
- 五种模式全部支持 `多图联合 / 批量套模板`。
- 默认模型切换到 Gemini 3.1 组合。
- 保持历史记录、模板中心、设置页、飞书同步链路可用。

### 4.3 发布目标

- 绿色包、ZIP、绿色安装包、Inno 安装器全部能生成。
- 安装器文件名、目录名、桌面快捷方式与项目名一致。
- 新版本默认使用 `%LOCALAPPDATA%\Picture-creation\data`，同时兼容旧数据目录。

## 5. 范围

### In scope

- Next.js 本地化 AI 生图工作台
- 五种生图模式与通用参数面板
- 多图联合 / 批量套模板 双语义
- 历史记录、审核导出、模板中心、设置页
- 飞书同步能力
- Windows 打包、安装器与发布文档
- GitHub 仓库初始化与对外 README

### Out of scope

- 云端 SaaS 化部署
- 多租户权限系统
- 移动端原生 App
- 模型训练与 LoRA 管理

## 6. 功能要求

1. 系统必须以 `Picture-creation` 作为统一应用名、仓库名和发布名。
2. 所有创建模式必须提供显式的 `多图联合 / 批量套模板` 切换。
3. `提示词模式` 必须支持不上传图片直接提交。
4. `参考图复刻` 必须支持原图区与参考图区共享 Gemini 3.1 的 14 张单次输入上限。
5. 创建页必须展示当前图片预览索引 `X/X`。
6. 发布脚本必须输出 `picture-creation` 系列目录、ZIP 与安装器。
7. 程序必须优先使用新数据目录，但能自动识别旧 `Commerce-Image-Studio` 数据。
8. README、使用说明、PRD、版本说明必须可直接交付给协作者使用。

## 9. 风险与约束

- 当前源码目录位于上级 Git 工作区内，不能直接拿上级仓库 remote 作为本项目独立仓库。
- 本地如有运行中的 `.next/standalone/server.js`，会阻塞重新构建与打包。
- 改名后若不保留旧数据兼容，会影响已有本地使用者迁移。
- 新旧文档并存时容易造成协作者误读，因此顶层入口必须只指向 `Picture-creation` 文档。

## 10. 发布标准

1. 版本号为 `0.7.0`。
2. GitHub 仓库名为 `Picture-creation`。
3. 安装器名为 `PICTURE-CREATION-WINDOWS-0.7.0.exe`。
4. 顶层 README 正确展示当前截图、文档入口和发布产物。
5. `Readme/` 中至少包含使用说明、PRD、版本说明和多图联合专题文档。
