# PRD-Picture-creation

对应对象：`Picture-creation`

> Owner: AEboli

## 1. 一句话概述

`Picture-creation` 是一个面向电商团队的本地化 AI 生图工作台，用于把“上传原图 / 组织参考图 / 批量生成 / 审核导出 / 品牌规则管理 / Windows 交付”收口到同一个桌面化工作流里。

## 2. 背景

电商团队在主图、详情页、活动图、A+ 图和多市场适配中，常见痛点是：

- Prompt、模板、品牌规则和设置项分散，难以复用。
- 同一套参数既需要“多图联合创作”，也需要“批量套同模板”。
- 成果审核、历史回看、导出与协作同步缺乏闭环。
- 本地安装交付和源码公开时，文档口径容易落后于真实页面状态。

`v0.9.0` 的核心目标，是把已经可用的工作台继续收敛到“个人浏览器配置可直接驱动生成、提示词策略可维护、配置故障可快速定位”的版本状态。

## 3. 目标用户

| 用户 | 诉求 |
| --- | --- |
| 电商运营 | 快速批量生成可上架、可投放的商品素材 |
| 设计支持 | 复用模板与品牌规则，减少重复劳动 |
| 商品团队 | 按品类、平台、市场稳定产出成套图片 |
| 维护者 | 能打包、能装机、能迁移、能定位问题 |

## 4. 产品目标

### 4.1 交付目标

- 对外 README、使用说明、PRD、版本说明全部同步到当前真实产品状态。
- 在 GitHub 上以 `v0.9.0` 发布源码、标签和版本说明。
- 保持安全发布口径，避免把本地数据、密钥或诊断目录带进源码仓库。

### 4.2 功能目标

- 保留五种创建模式与双生成语义。
- 主导航固定为 `总览 / 创作台 / 历史记录 / 设置 / 品牌库`。
- 设置页升级为紧凑控制中心，统一使用 `全局连接测试` 与 `保存全部设置`。
- 创建页生成、创建页智能体与 API 测试复用同一套浏览器本地 API 配置。
- 标准模式、套图模式、亚马逊 A+ 模式的图型提示词可在设置页编辑并进入生成链路。
- 侧栏 API / 飞书状态可直接打开对应快捷配置抽屉。
- 品牌库独立为一级页面；旧模板中心继续维持退役状态。
- 任务详情与 reference-remix 缺省文案的展示更清晰，减少歧义。

### 4.3 发布目标

- 绿色包、ZIP、绿色安装包、Inno 安装器全部能按 `0.9.0` 正常生成。
- 安装器文件名、目录名、桌面快捷方式与项目名一致。
- 安全发布包默认完成数据库脱敏与资产清空处理。

## 5. 范围

### In scope

- Next.js 本地化 AI 生图工作台
- 五种生图模式与通用参数面板
- 多图联合 / 批量套模板 双语义
- 历史记录、审核导出、品牌库、设置中心
- 飞书同步能力
- 创建页轻量智能体入口
- 浏览器本地 API 配置与 temporary provider 覆盖
- 图型提示词设置与 provider 生成链路注入
- 侧栏 API / 飞书快捷配置入口
- Windows 打包、安装器与 GitHub 发布文档

### Out of scope

- 云端 SaaS 化部署
- 多租户权限系统
- 移动端原生 App
- 旧模板中心重新上线
- 将创建页智能体升级为独立多轮产品

## 6. 功能要求

1. 系统必须以 `Picture-creation` 作为统一应用名、仓库名和发布名。
2. 所有创建模式必须继续提供显式的 `多图联合 / 批量套模板` 切换。
3. `提示词模式` 必须支持不上传图片直接提交。
4. `参考图复刻` 必须支持原图区与参考图区共享 Gemini 3.1 的 14 张单次输入上限。
5. 设置页必须提供统一的全局连接测试和整页保存动作。
6. 品牌库必须作为独立页面暴露，而不是继续混在设置页里。
7. `/templates` 不再作为可见主入口，退役口径必须保持一致。
8. 发布脚本必须继续输出 `picture-creation` 系列目录、ZIP 与安装器。
9. 程序必须优先使用新数据目录，但能自动识别旧 `Commerce-Image-Studio` 数据。
10. README、使用说明、PRD、版本说明必须可直接交付给协作者使用。
11. 创建页生成请求必须携带浏览器本地 API 设置，避免只依赖服务端保存的 API Key。
12. 创建页智能体必须复用浏览器本地文本模型配置；OpenAI 兼容 provider 必须走 Responses API。
13. 设置页必须提供可折叠的图型提示词编辑区，并保存到数据库配置中。
14. 生成链路必须把解析后的图型提示词传入 Gemini 与 OpenAI provider 的提示词构建过程。
15. 侧栏 API / 飞书状态必须可触发对应单区块快捷配置抽屉。

## 9. 风险与约束

- 当前仓库没有默认配置远端，需要先接好 GitHub 仓库再完成正式发布。
- 本地如有运行中的 `.next/standalone/server.js`，会阻塞重新构建与打包。
- 更名后若不保留旧数据兼容，会影响已有本地使用者迁移。
- 文档和截图若不及时更新，协作者会继续按旧模板中心、旧设置页或服务端 API Key 口径理解产品。
- 浏览器本地 API 设置不会跟随源码同步；协作者需要在自己的浏览器中重新填写密钥。

## 10. 发布标准

1. 版本号为 `0.9.0`。
2. GitHub 仓库名为 `Picture-creation`。
3. 安装器名为 `PICTURE-CREATION-WINDOWS-0.9.0.exe`。
4. 顶层 README 正确展示当前版本、文档入口与发布产物。
5. `Readme/` 中至少包含使用说明、PRD、版本说明和多图联合专题文档。
6. 安全发布口径明确排除 `.env`、本地数据库、运行产物与 `.learnings/`。
