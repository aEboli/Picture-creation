# PRD-多图联合生成语义
对应对象：Picture-creation 创建页多图输入语义统一

## 一句话概述

将创建页“多张原图”的含义从单一的“逐张批量处理”升级为双语义系统：保留历史的 `批量套模板`，并新增默认启用的 `多图联合`，使产品真正对齐 Gemini 3.1 Flash Image 的多图单次请求能力。

## 背景

项目原有逻辑里，“多张原图”实际表示批量逐张拆单；而 Gemini 3.1 Flash Image 已支持最多 14 张图共同参与单次生成。两者语义不一致，导致：

- 用户容易把“上传多张图”误解为“同一次生成会联合理解多张图”
- 前端计数、后端任务拆分和模型能力说明口径不一致
- 参考图复刻的“原图区 / 参考图区”与模型单次输入上限的关系不够清晰

## 目标

- 让五种创建模式都同时支持：
  - `多图联合`
  - `批量套模板`
- 默认值切到 `多图联合`，但不破坏旧任务与旧工作流
- 明确 `14 张图` 是单次请求上限，而不是整批任务上限

## 需求

1. 创建页必须在原图区显式提供“生成方式”切换器，值为 `joint` / `batch`。
2. 五种模式都必须支持两种生成方式，不能只在单一模式下生效。
3. `多图联合` 下，多张原图必须以平级输入身份参与同一次请求。
4. `批量套模板` 下，多张原图必须保持逐张拆分请求的历史行为。
5. 提示词模式必须继续支持 `0 张图 + 纯提示词`。
6. 参考图复刻必须保留“原图区 + 参考图区”双区结构，不合并为单区。
7. 参考图复刻在 `多图联合` 下，原图区与参考图区必须共享当前模型的单次请求总输入上限。
8. 创建页“实际生成数量”必须按当前生成方式切换计算公式。
9. 表单 payload、任务持久化、任务构建、任务执行链路必须打通 `generationSemantics` 字段。
10. 历史任务必须按 `batch` 兜底，避免重试旧任务时行为变化。

## 成功标准

- 创建页切换 `多图联合 / 批量套模板` 后，计数、校验、提交流程同步变化
- `joint` 下多张原图会进入同一次 Gemini 请求
- `batch` 下多张原图继续拆成多条独立任务
- `reference-remix` 下可以正确处理“原图区多图联合 + 参考图区共享上限”

## 约束

- 默认图片模型：`gemini-3.1-flash-image-preview`
- 默认文本模型：`gemini-3.1-flash-lite-preview`
- `14 张图` 解释为“单次请求总输入上限”
- 当前版本不引入“手动给每张图标角色”的重交互
